# Phase 2: Clergy Identity & Personal Benchmarking -- Design

**Date:** 2026-03-31
**Status:** Draft
**Scope:** Clergy self-identification, personalized compensation benchmarking, and neutral parish context data
**Depends on:** Phase 1 (CPG Data Integration & Database Consolidation)

## Problem

Vocation Hub+ shows the same data to every visitor. Clergy browsing open positions have no way to see how a position compares to their current situation -- compensation, parish size, geography, career trajectory. The data to power these comparisons exists in the DB (17,580 clergy records, 7,121 clergy positions, 194 compensation records, 67,957 parochial records) but nothing connects a visitor to their clergy record.

## Solution

Add a lightweight identity system that lets clergy claim their record via a bookmarkable URL token, then use that identity to show personalized compensation benchmarks, career context, and position comparisons. Also add neutral parish context data (clergy tenure, attendance/giving trends) visible to all users.

## Identity System

### Token Design

Pre-generated at build time for all clergy in the DB. Each clergy record gets a random 12-character URL-safe token stored in a `clergy_tokens` table.

```sql
CREATE TABLE clergy_tokens (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  token TEXT NOT NULL UNIQUE,
  clergy_guid TEXT NOT NULL REFERENCES clergy(guid),
  claimed_at DATETIME DEFAULT (datetime('now'))
);
```

Tokens are deterministic per build: generated by hashing the clergy GUID with an HMAC secret (stored as an environment variable / GitHub Actions secret, `CLERGY_TOKEN_SECRET`). This means the same clergy member gets the same token across rebuilds (bookmarked URLs stay valid) but the token can't be derived from the GUID without knowing the secret. The full token-to-data mapping is exported as `clergy-tokens.json` (~2-3MB gzipped) and a lightweight search index as `clergy-search-index.json`.

### Claim Flow

1. User clicks "This is me" in the navigation bar (visible when no active token).
2. `/claim` page shows a text input: "Enter your name."
3. As they type (debounced, 3+ characters), fuzzy-matched clergy cards appear showing:
   - Full name
   - Current position and parish
   - Diocese
   - Ordination year
   - City, state
4. User clicks "This is me" on the correct card.
5. Browser redirects to `/?me=<token>`. Token is also saved to localStorage for subsequent visits.

Search uses Fuse.js (already a project dependency) against `clergy-search-index.json`.

### URL Behavior

| URL | Behavior |
|-----|----------|
| `/?me=abc123` | Main positions page with inline personal context |
| `/me?me=abc123` | Personal dashboard |
| `/me` (no token, but localStorage has one) | Dashboard using cached token |
| `/me` (no token, no cache) | Redirect to `/claim` |
| `/claim` | Claim page; if already claimed, shows "Switch identity" option |

Token persists in localStorage so the `?me=` parameter can be dropped from subsequent navigation within the site. The bookmarkable link with `?me=` lets the user return from any device.

### Privacy Considerations

All data in the token map comes from the ECDPlus public directory (unauthenticated API) and CPG published compensation reports (aggregated by diocese, not individual). No private data is exposed. The tokens are opaque -- knowing a token lets you see one clergy member's public directory info plus diocese-level compensation medians, which is the same information available by browsing the ECDPlus directory directly.

## Personal Dashboard (`/me`)

### Layout

Summary bar at top with key stats, followed by three tabbed sections.

### Summary Bar

- Clergy name and current position/parish
- Diocese and ordination year
- Total estimated compensation (if derivable from current position data)
- Diocese compensation percentile
- Years of experience (from ordination date)

### Tab 1: Compensation

Multi-dimensional benchmark showing where the clergy member falls relative to peers across several axes:

- **Diocese median** -- their diocese's median clergy compensation vs. national
- **ASA bucket median** -- compensation for churches their size
- **Position type median** -- rector vs. associate rector vs. interim, etc.
- **Experience bracket median** -- compensation for their years of experience
- **Gender-specific median** -- if applicable, male/female diocesan medians

Visualized as a radar/spider chart (Recharts) with a data table below for exact numbers. Each axis shows "your position" vs. "median for this dimension."

Data source: `compensation_diocesan`, `compensation_by_asa`, `compensation_by_position`, `compensation_by_experience` tables.

### Tab 2: Career Timeline

Vertical timeline of the clergy member's position history from `clergy_positions`:

- Each position shows: title, parish name, diocese, start/end dates, duration
- Current position highlighted
- Ordination date as the timeline start
- If parish is in our DB, link to its data (ASA, giving, etc.)

### Tab 3: Recommended Positions

Open positions filtered and sorted by relevance to the claimed clergy member. Scoring factors:

- Same position type as current role (or logical next step: associate -> rector)
- Similar ASA range to current parish
- Similar compensation range
- Same or adjacent state/region
- Diocese where they've previously served

Each recommended position links to the main position table with `?me=` preserved so inline context is visible. Scoring weights are tunable at implementation time; the factors above are listed in rough priority order.

## Inline Personal Context

When a claimed user browses positions, expanded position cards show a "Personal Context" section with up to 18 comparison data points. Each point gracefully hides when data is unavailable.

### Compensation Comparisons

| Data Point | Computation |
|------------|-------------|
| Diocese median comparison | This position's diocese median vs. your diocese median (% difference) |
| Estimated total comp comparison | This position's estimated comp vs. yours |
| ASA-bucket median comparison | Median for churches this size vs. churches your size |
| Position-type median comparison | Median for this role type vs. your role type |

### Parish Profile Comparisons

| Data Point | Computation |
|------------|-------------|
| ASA comparison | This parish's ASA vs. your parish's ASA |
| Plate & pledge comparison | This parish's annual giving vs. yours |
| Membership comparison | This parish's membership vs. yours |
| Operating revenue comparison | If available for both parishes |
| Parish context | Clergy count and avg tenure (shown to all users) |

### Geographic & Practical

| Data Point | Computation |
|------------|-------------|
| Distance from current parish | Haversine distance using lat/lng from both parishes |
| Cost of living proxy | Census median household income (yours vs. theirs) |
| Stipend-to-local-income ratio | How far the pay goes locally |
| Relocation indicator | "Same diocese" / "Different diocese, same state" / "Relocation required" |

### Career Fit

| Data Point | Computation |
|------------|-------------|
| Experience match | Your years of experience vs. typical experience at this parish size |
| Position type match | Whether this is the same role type or a common career progression |
| Diocese familiarity | Whether you've served in this diocese before (from position history) |
| Time since last move | Your current tenure vs. average tenure for your role type |

### People Connections

| Data Point | Computation |
|------------|-------------|
| Mutual diocese connections | Clergy who've served in both your current diocese and this position's diocese |
| Current clergy | Who's currently at this parish (name, title, tenure) -- already displayed but highlighted if you share ordination year or training diocese |

### Implementation

All comparisons are computed client-side in `web/src/lib/personal-context.ts` as pure functions. The component (`PersonalContext.tsx`) receives the claimed user's data object and the position object, calls the computation functions, and renders results that have non-null values.

## Parish Context (All Users)

Neutral factual data about each parish, shown on every expanded position card regardless of claimed identity. Replaces the removed "parish stability scoring" without editorial judgment.

### New Fields on Enriched Positions

```typescript
parish_context: {
  clergy_count_10yr: number;        // clergy who served here in last 10 years
  avg_tenure_years: number;         // average tenure of those clergy
  current_clergy_count: number;     // clergy currently assigned
  attendance_trend: 'growing' | 'declining' | 'stable' | null;
  attendance_change_pct: number | null;   // percent change over 5 years
  giving_trend: 'growing' | 'declining' | 'stable' | null;
  giving_change_pct: number | null;
  membership_trend: 'growing' | 'declining' | 'stable' | null;
  membership_change_pct: number | null;
  latest_operating_revenue: number | null;
  years_of_data: number;            // parochial report years available
}
```

### Presentation

A "Parish Context" section in the expanded position detail, between "Diocese Context" and "Community Context." No color coding, no scores, no badges. Factual statements only:

- "3 clergy in the past 10 years (avg tenure: 2.8 years)"
- "Average Sunday Attendance: 145 to 128 over 5 years (-12%)"
- "Plate & Pledge: $285K to $310K over 5 years (+9%)"
- "Membership: 312 to 290 over 5 years (-7%)"
- "Based on X years of parochial report data"

Trend direction is stated factually with start and end values plus percentage. The user draws their own conclusions.

### Computation

Performed at build time in `enrich-positions-v2.js` by querying `clergy_positions` and `parochial_data` tables for each matched parish.

## Build Pipeline

### New Script: `generate-clergy-tokens.js`

CommonJS module. Reads from DB:
- `clergy` table for identity info
- `clergy_positions` table for career history
- `compensation_*` tables for benchmark data
- `parishes` + `parochial_data` for current parish context

Outputs:
- `web/public/data/clergy-tokens.json` -- Full token-to-data map
- `web/public/data/clergy-search-index.json` -- Lightweight search array

### Modified: `enrich-positions-v2.js`

Adds `parish_context` computation for all positions:
- Query `clergy_positions` for clergy history at matched parish
- Query `parochial_data` for attendance/giving/membership trends
- Compute trend direction and percentage change

### Pipeline Order

1. `generate-clergy-tokens.js` (independent of enrichment; can run in parallel)
2. `enrich-positions-v2.js` (adds parish_context to all positions; independent of token generation)
3. `next build` (consumes all JSON outputs)

### Workflow Integration

Add `generate-clergy-tokens.js` to:
- `data-refresh.yml` (after DB is populated, before enrichment)
- `scrape.yml` (after enrichment, to pick up new position matches)

## Not In Scope

- Server-side rendering or API endpoints (stays fully static)
- Email notifications for new matching positions
- Clergy-to-clergy messaging
- Diocesan compensation PDF parsing improvements (Phase 1 coverage is sufficient)
- Career trajectory predictions or vacancy forecasting (Phase 2 preview items deferred to Phase 3)
